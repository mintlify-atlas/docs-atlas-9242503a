## Overview

The `process` function is Splat's AI engine that analyzes Python error tracebacks and provides structured, actionable debugging advice. It uses Groq's `llama3-70b-8192` model to understand errors and suggest specific code fixes.

## Function Signature

```python
def process(
    traceback_message: str, 
    original_error_information: str, 
    context: str
) -> object
```

## Parameters

<ParamField path="traceback_message" type="str" required>
  The complete error traceback captured from the failed execution. This includes the full stack trace showing where the error occurred.
  
  **Example:**
  ```
  File "/path/to/script.py", line 42
    print(hello
          ^
  SyntaxError: '(' was never closed
  ```
</ParamField>

<ParamField path="original_error_information" type="str" required>
  String representation of the original error, typically from `str(error)` on the exception object.
</ParamField>

<ParamField path="context" type="str" required>
  The collected file contents from the error stack and related files. This provides the LLM with full context about the codebase to make informed suggestions.
  
  Generated by `run_mock_repopack()` from the relational module.
</ParamField>

## Return Value

<ResponseField name="response" type="str">
  A JSON string containing structured error analysis with the following schema:
  
  ```json
  {
    "where": {
      "repository_path": "<absolute path to repository>",
      "file_name": "<name of file containing error>",
      "line_number": "<line number where error occurred>"
    },
    "what": {
      "error_type": "<specific Python error type>",
      "description": "<concise explanation of error>"
    },
    "how": {
      "error_origination": "<line number where error originated>",
      "suggested_code_solution": "<code snippet to fix the error>"
    }
  }
  ```
</ResponseField>

## AI Model Configuration

<ParamField path="model" type="str" default="llama3-70b-8192">
  The Groq model used for analysis. Currently uses `llama3-70b-8192` (Llama 3 70B parameter model with 8K context window).
</ParamField>

<ParamField path="response_format" type="dict" default='{"type": "json_object"}'>
  Forces the model to return valid JSON output for structured parsing.
</ParamField>

## System Prompts

The function uses a carefully crafted multi-part system prompt to guide the AI:

### Prompt 1: Role Definition
```
You are an expert software debugging assistant specializing in Python error 
analysis. Your task is to analyze error tracebacks and provide structured, 
actionable advice. Follow these steps precisely:
```

### Prompt 2: Analysis Steps
```
1. Analyze the provided error traceback and original error message.
2. Identify the source of the error within the repository structure.
3. Explain the error concisely in natural language.
4. Provide specific, actionable suggestions for resolving the error.
5. Format your response as a JSON object with the exact structure specified below.
```

### Prompt 3: JSON Structure
Defines the exact schema for the `where`, `what`, and `how` response fields.

### Prompt 4: Constraints
```
- Provide ONLY the JSON object as your response. Do not include any other text.
- Ensure all JSON keys are exactly as specified.
- The 'suggested_code_solution' should be a valid Python code snippet without explanation.
- If multiple errors exist, focus on the most critical one that likely caused the others.
- Do not use placeholders in your response. Provide specific, contextual information based on the given error.
- Only fix the line from the line number given that is causing the error.
```

## Usage Examples

### Basic Error Analysis

```python
from process.process import process
import json

traceback = '''
File "/Users/dev/project/main.py", line 15
  print(result
        ^
SyntaxError: '(' was never closed
'''

error_info = "Command '['python3', 'main.py']' returned non-zero exit status 1."

context = """
File: /Users/dev/project/main.py
Content:
def calculate():
  result = 5 + 10
  print(result
  return result
"""

# Get AI analysis
response = process(traceback, error_info, context)
analysis = json.loads(response)

print(f"Error Type: {analysis['what']['error_type']}")
print(f"Location: {analysis['where']['file_name']}:{analysis['where']['line_number']}")
print(f"Fix: {analysis['how']['suggested_code_solution']}")
```

### Complete Pipeline Integration

```python
from relational import relational_error_parsing_function
from process.process import process
import json

# Step 1: Execute and capture error
traceback, error_info, context = relational_error_parsing_function(
    ['python3', 'app.py'],
    flag='-r'  # Include dependencies
)

# Step 2: Analyze with AI
if traceback:
    analysis_json = process(traceback, error_info, context)
    analysis = json.loads(analysis_json)
    
    # Step 3: Display results
    print(f"""
    Error Analysis:
    ===============
    File: {analysis['where']['file_name']}
    Line: {analysis['where']['line_number']}
    Type: {analysis['what']['error_type']}
    
    Description:
    {analysis['what']['description']}
    
    Suggested Fix:
    {analysis['how']['suggested_code_solution']}
    """)
```

## Environment Setup

<Note>
  The function requires a Groq API key to be set in the environment. Splat uses `python-dotenv` to load this from a `.env` file.
</Note>

**Required Environment Variable:**

```bash
# .env file
API=your_groq_api_key_here
```

The API key is accessed via:
```python
client = Groq(api_key=os.getenv("API"))
```

## Response Format

The JSON response is guaranteed to follow this structure:

<ResponseField name="where" type="object">
  Location information for the error
  
  <ResponseField name="where.repository_path" type="string">
    Absolute path to the repository root
  </ResponseField>
  
  <ResponseField name="where.file_name" type="string">
    Name of the file containing the error
  </ResponseField>
  
  <ResponseField name="where.line_number" type="string">
    Line number where the error occurred
  </ResponseField>
</ResponseField>

<ResponseField name="what" type="object">
  Description of the error
  
  <ResponseField name="what.error_type" type="string">
    Specific Python error type (e.g., "SyntaxError", "TypeError", "NameError")
  </ResponseField>
  
  <ResponseField name="what.description" type="string">
    Human-readable explanation of what went wrong
  </ResponseField>
</ResponseField>

<ResponseField name="how" type="object">
  Solution information
  
  <ResponseField name="how.error_origination" type="string">
    Line number where the error originated (may differ from where it was caught)
  </ResponseField>
  
  <ResponseField name="how.suggested_code_solution" type="string">
    Executable Python code snippet that fixes the error
  </ResponseField>
</ResponseField>

## Best Practices

1. **Provide Rich Context**: The more complete your `context` parameter, the better the AI's suggestions will be
2. **Use Recursive Mode**: When calling `relational_error_parsing_function`, use `-r` flag for complex errors that may involve multiple files
3. **Parse JSON Safely**: Always use `json.loads()` to parse the response and handle potential parsing errors
4. **Handle API Errors**: Wrap calls in try-except blocks to handle Groq API failures gracefully

## Dependencies

- `groq`: Groq Python client library
- `python-dotenv`: For loading environment variables
- `terminalout.terminal`: Custom terminal output formatting (used for display)
