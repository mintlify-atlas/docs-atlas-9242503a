## Overview

This page demonstrates how splat handles various types of Python errors in real-world scenarios. Each example shows the complete workflow from error to solution.

## Example 1: Syntax Error - Unclosed Parenthesis

### The Problem

You have a simple Python script with a syntax error:

```python test.py
print("Starting the application...")
print("Hello, World!"
print("Application finished.")
```

### Running Splat

```bash
splat squash "python3 test.py"
```

<Accordion title="View Terminal Output">
```
File "test.py", line 2
    print("Hello, World!"
                          ^
SyntaxError: '(' was never closed

Analyzing error with AI...
```
</Accordion>

### Splat's Response

```json
{
  "where": {
    "repository_path": "/home/user/project",
    "file_name": "test.py",
    "line_number": "2"
  },
  "what": {
    "error_type": "SyntaxError",
    "description": "Opening parenthesis on line 2 was never closed. Python expects a closing parenthesis before the line ends."
  },
  "how": {
    "error_origination": "2",
    "suggested_code_solution": "print(\"Hello, World!\")"
  }
}
```

### The Fix

Update line 2:

```python test.py
print("Starting the application...")
print("Hello, World!")  # Added closing parenthesis
print("Application finished.")
```

## Example 2: Name Error - Undefined Variable

### The Problem

A script references a variable that doesn't exist:

```python calculator.py
def calculate_total(items):
    total = 0
    for item in items:
        total += item.price
    return totla  # Typo: 'totla' instead of 'total'

result = calculate_total([{"price": 10}, {"price": 20}])
print(f"Total: {result}")
```

```bash
splat squash "python3 calculator.py"
```

## Example 4: Type Error with Relational Mode

### The Problem

A complex project with multiple files has a type mismatch:

**models.py**
```python
class User:
    def __init__(self, user_id, name):
        self.user_id = user_id
        self.name = name
    
    def get_id(self):
        return self.user_id
```

**database.py**
```python
from models import User

def fetch_user(user_id):
    # Simulating database fetch
    return {"id": user_id, "name": "John Doe"}  # Returns dict, not User object
```

**services.py**
```python
from database import fetch_user

def get_user_name(user_id):
    user = fetch_user(user_id)
    return user.get_id()  # Error: dict has no method 'get_id'
```

**main.py**
```python
from services import get_user_name

if __name__ == "__main__":
    name = get_user_name(123)
    print(f"User: {name}")
```

### Running Splat (Basic Mode)

```bash
splat squash "python3 main.py"
```

Basic mode might not have enough context to understand the issue fully.

### Running Splat (Relational Mode)

```bash
splat squash -r "python3 main.py"
```

<Accordion title="View Error Traceback">
```
Traceback (most recent call last):
  File "main.py", line 4, in <module>
    name = get_user_name(123)
  File "services.py", line 5, in get_user_name
    return user.get_id()
AttributeError: 'dict' object has no attribute 'get_id'
```
</Accordion>

### Splat's Response (with -r flag)

```json
{
  "where": {
    "repository_path": "/home/user/project",
    "file_name": "services.py",
    "line_number": "5"
  },
  "what": {
    "error_type": "AttributeError",
    "description": "The fetch_user() function in database.py returns a dictionary, but services.py expects a User object with a get_id() method. There's a type mismatch between what database.py returns and what services.py expects."
  },
  "how": {
    "error_origination": "5",
    "suggested_code_solution": "# Option 1: Fix database.py to return User object:\nfrom models import User\n\ndef fetch_user(user_id):\n    data = {\"id\": user_id, \"name\": \"John Doe\"}\n    return User(data['id'], data['name'])\n\n# Option 2: Fix services.py to handle dict:\ndef get_user_name(user_id):\n    user = fetch_user(user_id)\n    return user['id'] if isinstance(user, dict) else user.get_id()"
  }
}
```

<Note>
With the `-r` flag, splat analyzed all four files and understood the complete data flow, enabling it to identify the type mismatch between `database.py` and `services.py`.
</Note>

```bash
splat squash "python3 stats.py"
```

</Accordion>

### Splat's Response

```json
{
  "where": {
    "repository_path": "/home/user/project",
    "file_name": "stats.py",
    "line_number": "4"
  },
  "what": {
    "error_type": "ZeroDivisionError",
    "description": "Cannot divide by zero. The function is called with an empty list, so len(numbers) is 0, causing division by zero."
```

## Comparison: Basic vs Relational Mode

Here's a side-by-side comparison showing when to use each mode:

<Tabs>
  <Tab title="Simple Error (Basic Mode)">
    ### Scenario: Single File Syntax Error
    
    ```python script.py
    def greet(name):
        print(f"Hello, {name}"  # Missing closing parenthesis
    
    greet("World")
    ```
    
    **Command:**
    ```bash
    splat squash "python3 script.py"
    ```
    
    **Result:** Fast response, accurate fix. Basic mode is sufficient.
    
    ✅ **Use basic mode** - Error is isolated to one file
  </Tab>
  
  <Tab title="Complex Error (Relational Mode)">
    ### Scenario: Multi-File Import Issue
    
    **Project Structure:**
    ```
    project/
    ├── main.py (imports services)
    ├── services.py (imports models, database)
    ├── models.py (imports validators)
    ├── database.py (imports config)
    ├── validators.py
    └── config.py
    ```
    
    **Error:** Type mismatch between `database.py` and `services.py`
    
    **Command:**
    ```bash
    splat squash -r "python3 main.py"
    ```
    
    **Result:** Splat analyzes all 6 files, understands the complete data flow, and identifies the root cause.
    
    ✅ **Use relational mode** - Error involves multiple interconnected files
  </Tab>
</Tabs>
